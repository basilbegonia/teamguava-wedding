## Google Sheet Structure

One Google Sheet with two tabs:

### Tab 1: `Guests`
| Column            | Notes                                                                                          |
| ----------------- | ---------------------------------------------------------------------------------------------- |
| name              | Full name — you populate this                                                                  |
| email             | Guest email — you populate this                                                                |
| token             | UUID — generated by a script or manually with `crypto.randomUUID()`                           |
| visited_at        | Written by the app on first valid token use                                                    |
| rsvp_submitted_at | Written by the app on RSVP submission (set for every member of the party at the same time)     |
| party_id          | Optional. Guests sharing the same non-empty value are in the same household/party. Leave blank for solo guests. |

### Tab 2: `RSVPs`
| Column       | Notes                                                                        |
| ------------ | ---------------------------------------------------------------------------- |
| guest_token  | FK to Guests.token — one row per guest, even in a party                      |
| name         | Pre-filled from Guests tab (not session)                                     |
| attendance   | `ceremony_and_reception` / `ceremony_only` / `not_coming`                    |
| dietary      | Free text                                                                    |
| allergies    | Free text                                                                    |
| transport    | Free text                                                                    |
| submitted_at | ISO timestamp                                                                |

## Authentication — Magic Link Flow

1. You populate the `Guests` sheet with name, email, UUID token, and optional `party_id` per guest
2. Invite URL format: `https://teamguava.life/invite/[token]`
3. For households: every member gets their own token and unique invite URL, but they all share a `party_id`
4. You email/message each guest their unique invite URL manually (no email sending in v1)
5. When a guest visits `/invite/[token]`:
   - App reads and caches the Guests sheet (5-minute TTL via `unstable_cache`)
   - Validates the token exists
   - Sets an httpOnly JWT session cookie (30 days) containing `{ name, token }`
   - Writes `visited_at` timestamp back to the Guests sheet **for that guest only**
   - Redirects to `/`
6. Next.js middleware checks for valid session cookie on ALL routes
7. No session → redirect to `/invite/error`
8. Invalid/unknown token → render friendly error page ("hmm, this link doesn't look right")

## Middleware

```typescript
// middleware.ts — protects all routes except /invite/[token] and /invite/error
import { NextRequest, NextResponse } from 'next/server'
import { jwtVerify } from 'jose'

const PUBLIC_PATHS = ['/invite']

export async function middleware(req: NextRequest) {
  const { pathname } = req.nextUrl
  if (PUBLIC_PATHS.some(p => pathname.startsWith(p))) return NextResponse.next()

  const token = req.cookies.get('guava_session')?.value
  if (!token) return NextResponse.redirect(new URL('/invite/error', req.url))

  try {
    await jwtVerify(token, new TextEncoder().encode(process.env.SESSION_SECRET!))
    return NextResponse.next()
  } catch {
    return NextResponse.redirect(new URL('/invite/error', req.url))
  }
}

export const config = {
  matcher: ['/((?!_next/static|_next/image|favicon.ico).*)'],
}
```

## Google Sheets Helper (lib/sheets.ts)

Use `googleapis` npm package with a JWT auth client from the service account credentials.

### Functions

| Function | Description |
|---|---|
| `getGuests()` | Reads Guests tab (`A:F`), returns `Guest[]`. Cached via `unstable_cache` at 5-min TTL, tag `guests`. |
| `getParty(token)` | Returns all guests whose `party_id` matches the given token's guest. If the guest has no `party_id`, returns `[guest]` (solo party). |
| `markVisited(token)` | Finds guest row by token, writes current timestamp to `visited_at` (column D). Invalidates `guests` cache. |
| `markRsvpSubmitted(tokens[])` | Writes current timestamp to `rsvp_submitted_at` (column E) for all given tokens in one `batchUpdate` call. Invalidates `guests` cache. |
| `getRSVP(token)` | Returns the existing RSVP row for a single guest token, or `null`. |
| `getPartyRSVPs(tokens[])` | Reads RSVPs tab once, returns `(RSVPData \| null)[]` in the same order as the input tokens. |
| `appendRSVP(data)` | Appends a single new row to the RSVPs tab. |
| `updateRSVP(token, data)` | Finds existing RSVP row by token and overwrites it; falls back to `appendRSVP` if not found. |
| `submitPartyRSVPs(rsvps[])` | Batch-writes all party RSVPs (one `batchUpdate` for updates + one `append` for new rows), then calls `markRsvpSubmitted` for all tokens. |

### Interfaces

```ts
interface Guest {
  name: string
  email: string
  token: string
  visited_at: string
  rsvp_submitted_at: string
  party_id: string   // empty string for solo guests
}

interface RSVPData {
  guest_token: string
  name: string
  attendance: 'ceremony_and_reception' | 'ceremony_only' | 'not_coming'
  dietary: string
  allergies: string
  transport: string
  submitted_at: string
}
```

## Notes for Claude Code

- Build tasks sequentially — each task has a clear acceptance criterion before moving on
- Keep all secrets in env vars — never hardcode
- Google Sheets private key in env will have literal `\n` — handle with `.replace(/\\n/g, '\n')` when constructing the JWT auth client
- The Guests sheet cache must be invalidated after `markVisited()` and `markRsvpSubmitted()` writes — use `revalidateTag`
- All pages are mobile-first — test at 390px width
- Bilingual copy is intentional — don't "correct" Filipino phrases
- No plus-ones: the RSVP form does not have a guest count field; the party list is strictly what's in the Guests sheet
- The magic link is the only way in — there is no fallback password or login page
- **Party grouping**: `party_id` is optional. Solo guests (no `party_id`) go through the same RSVP flow — their "party" is just themselves. The RSVP API always validates that submitted tokens belong to the session owner's party to prevent guests from editing each other's RSVPs across parties.
- Each guest in a party gets their own unique invite URL. Any one of them can RSVP for the whole party. `visited_at` is only written for the individual who clicked the link; `rsvp_submitted_at` is written for all party members simultaneously when the RSVP is submitted.
- The `Guests!A:F` range must include the `party_id` header in row 1 of the sheet.
